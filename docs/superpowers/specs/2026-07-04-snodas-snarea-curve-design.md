# Design: SNODAS-derived snow depletion curves → PRMS `snarea_curve`

**Date:** 2026-07-04
**Status:** Approved design (pre-implementation)
**Source method:** Driscoll, Hay & Bock (2017), *Spatiotemporal Variability of Snow
Depletion Curves Derived from SNODAS for the Conterminous United States, 2004–2013*,
JAWRA 53(3):655–666. DOI: 10.1111/1752-1688.12520. Local copy:
[`docs/Snow_Depletion_Curves.md`](../../Snow_Depletion_Curves.md).

## 1. Goal

Produce the PRMS/NHM **`snarea_curve`** parameter (per-HRU snow-area depletion
curves) for a watershed fabric, derived from daily SNODAS SWE via the Driscoll
et al. (2017) methodology. Validate on the **Oregon** fabric, then run **CONUS
gfv2**.

The paper's Figure 7 — snow-covered area (SCA) sampled at normalized SWE =
1.0, 0.9, …, 0.0 — *is* the 11-point PRMS depletion curve, so the paper's
"representative SDC per HRU" maps directly onto the model parameter.

### Locked decisions

| Decision | Choice |
|---|---|
| Primary deliverable | PRMS `snarea_curve` (one curve per HRU) |
| Curve structure | One curve per HRU; `hru_deplcrv` is 1:1; plus a `sdc_status` flag column |
| Validation fabric | Oregon (VPU 17), then CONUS gfv2 |
| Study period | All available years, **2004–2024** (up to 21 melt seasons; paper had 9) |
| Fallback (HRUs failing selection) | Assign the **NHM default depletion curve**, flagged in `sdc_status` |
| Where the aggregation lives | **Standalone harness in gfv2-params** (borrowed pattern from nhf-spatial-targets; NOT a dependency), kept source-agnostic for future climate→fabric mapping |
| Aggregation & analysis cadence | Per **calendar** year (matches the SNODAS input files and preferred workflow). For seasonal snowpacks the melt season (peak → SWE=0) lies within one calendar year, so each year yields one seasonal SDC with no cross-year windowing; edge HRUs whose peak falls near a year boundary are flagged (§4 step 1) |

## 2. Inputs (all on disk)

- **Raw SNODAS daily SWE:** `…/nhf-datastore/snodas/daily/snodas_daily_<year>.nc`,
  2004–2024. CF-1.6, var `swe` (kg m⁻² = mm), **pre-projected to EPSG:5070**
  (1000 m NAD83 CONUS Albers), `_FillValue = -9999` over ocean / non-CONUS /
  Great Lakes.
- **Oregon fabric:** `{data_root}/oregon/fabric/model_layers 9.gpkg`, layer `nhru`,
  `id_feature = hru_id` (contiguous 1..16814). *Same underlying fabric* the
  nhf-spatial-targets SNODAS aggregate used, where it is keyed `nhm_id` — see
  §7 integration note.
- **Existing per-HRU daily SNODAS mean SWE (validation oracle, not a dependency):**
  `…/or-spatial-targets/data/aggregated/snodas/snodas_<year>_agg.nc`
  (`{time, nhm_id:16814}`, var `swe`) and the CONUS equivalent under
  `…/gfv2-spatial-targets/…` (`nat_hru_id:361471`). Used only to cross-check
  our standalone harness's SWE output.
- **Elevation / slope per HRU:** already generated by this repo's zonal pass
  (used for the optional topo/climate relation, and available for QA).

## 3. Architecture

### 3.0 Fabric independence (non-negotiable)

This work obeys the repo's single-source-of-truth rule: **process configs are
fabric-agnostic; every per-fabric input is read from the active fabric profile in
`configs/base_config.yml` via `require_config_key`**, using the
`{data_root}`/`{fabric}` (and `{vpu}` where relevant) placeholders — never a
literal path or a naming convention. Oregon is merely the first profile the
pipeline is *instantiated* against; the same `--fabric gfv2` run targets CONUS
with no code or process-config change.

Concretely:
- **New process configs** (`configs/aggregate/…`, and the `snarea_curve` entry)
  contain only placeholdered paths and source-adapter settings — no fabric names.
- **Per-fabric inputs** resolve from the profile: the fabric geopackage
  (`hru_gpkg`/`hru_layer`), the HRU id column (`id_feature`), and output dirs via
  `{data_root}/{fabric}/…`. All already exist in every profile.
- **New profile key — `snodas_dir`**: the raw SNODAS daily directory. Because the
  raw SNODAS datastore is shared across fabrics, it is defaulted once in the
  aggregate process config (pointing at the shared datastore) and is
  *overridable* per profile, matching how other shared inputs (e.g.
  `waterbody_gpkg`) are handled — resolved with `require_config_key`, not
  hardcoded in a builder.
- **The aggregation harness is fabric-agnostic:** a `SourceAdapter` describes a
  *source*, not a fabric; the driver receives `fabric_path`/`id_col` resolved
  from the profile at call time. The same harness serves any fabric and any
  future gridded source (e.g. climate).
- Any Oregon-specific path appearing in §2/§7 is the profile-*resolved* value
  shown for concreteness, not a literal that lands in code or a process config.

### 3.1 The three pieces

Three pieces, all in gfv2-params, each with a config block and a
`tests/test_*.py`:

```
raw SNODAS daily SWE (EPSG:5070, per calendar year)
      │
      ▼  Piece 1+2: aggregation harness + SNODAS adapter
   per-HRU daily {swe_mm, sca_frac}, one file per calendar year
      │
      ▼  Piece 3: snarea_curve builder (SDC math)
   snarea_curve (11×nhru) + hru_deplcrv (1:1) + sdc_status
      │
      ▼
   PRMS parameter output (Oregon → CONUS gfv2)
```

#### Piece 1 — general gridded-time-series → HRU aggregation harness

New subpackage `src/gfv2_params/aggregate/`. The **time-series counterpart** to
the existing static-raster `UserTiffData`/`ZonalGen` runners in
`src/gfv2_params/zonal_runners/`. Pattern borrowed from
`nhf-spatial-targets/src/nhf_spatial_targets/aggregate/{_adapter,_driver}.py`,
trimmed to gfv2-params conventions (no manifest/lineage/release machinery).

- **`SourceAdapter`** (frozen dataclass): `source_key`, `variables: tuple`,
  `source_crs`, `x_coord/y_coord/time_coord`, `files_glob`, `pre_aggregate_hook`,
  `stat_method` (validated against gdptools' STATSMETHODS), `output_cadence`.
- **Driver** (`aggregate_source`):
  1. Load fabric (reprojected to `weight_gen_crs`); build grid→polygon weights
     **once** with `WeightGen(output_file=…)`, cached to the weight dir
     (mirrors `run_build_weights`'s compute-once/cache/reuse idiom).
  2. For each input file (calendar year): open lazily, run `pre_aggregate_hook`,
     wrap in `UserCatData` (with `source_t_coord`), aggregate with
     `AggGen(stat_method=…, weights=<cached>)`.
  3. Write one per-HRU per-day output file per calendar year.
- Reuses gfv2-params' own `batching.py` for HRU batching; slots into the
  `derive_*`/config-flatten/`{placeholder}` orchestrator pattern.
- **Source-agnostic by design** — a future Daymet/climate adapter is just a new
  `SourceAdapter`.

#### Piece 2 — SNODAS adapter

A `SourceAdapter` instance:
- `variables = ("swe", "scov")`, `stat_method = "masked_mean"`,
  `source_crs = "EPSG:5070"`, `files_glob = "snodas_daily_*.nc"`.
- `pre_aggregate_hook`: re-assert the fill mask (`swe.where(swe > -9990)`), then
  derive `scov = (swe > 0)` as a float **carrying that NaN mask** (fill pixels
  stay NaN, not 0).
- `masked_mean` semantics give:
  - `swe` → area-weighted **mean SWE (mm)** per HRU per day (matches the
    existing oracle within tolerance),
  - `scov` → area-weighted mean of a 0/1 field = **snow-covered-area fraction**,
    the paper's SCA, with ocean/fill pixels excluded from the denominator
    (correct; the SDC selection drops water-dominated HRUs anyway).
- Output: per-HRU daily `{swe_mm, sca_frac}`, one file per calendar year, under
  `{data_root}/{fabric}/snodas/`.

#### Piece 3 — `snarea_curve` builder

Pure pandas/numpy over the per-HRU daily `{swe_mm, sca_frac}`. New builder module
+ config + test, following the repo's builder→merged-param convention. Algorithm
in §4.

## 4. SDC derivation algorithm (Piece 3)

**Per HRU, per calendar-year melt season:**

1. **Isolate the melt season within the calendar year.** For seasonal snowpacks
   peak → SWE=0 lies inside the year, so no cross-year read is needed. Guard the
   edge cases: SWE never returning to 0 (persistent snowfield/glacier) or a peak
   in the first/last days of the file — such HRU-years are excluded from that
   season's SDC and flagged (they feed selection criterion 6 / the default
   fallback if they dominate the record).
2. **Peak SWE:** `t_peak` = date of max SWE in the calendar year; `peak_swe` its value.
3. **Melt window:** `t_peak` → the first day on which SWE returns to 0.
4. **Remove reversals** (paper's method): within the melt window, any interval
   where SWE *increases* (post-peak snowfall) is a "reversal" — drop those days
   until SCA returns to its pre-increase value, then resume. Yields an idealized,
   monotonic melt. (Reversal count retained as a per-HRU diagnostic.)
5. **Normalize:** `swe_n = swe / peak_swe` (1 → 0); `sca_n = sca / sca_at_peak`
   (SCA normalized by its value at peak SWE — *judgment call, see §8*).
6. **Annual SDC:** interpolate `sca_n` at `swe_n = 1.0, 0.9, …, 0.0` → an 11-point
   vector for that season.

**Across the ≤21 annual SDCs for the HRU:**

7. **Median SDC:** elementwise median across seasons.
8. **Similarity** (Eq. 1): `Σ over seasons Σ over points |sca_n − median| / points`
   — a per-HRU scalar; higher = more inter-annual variability.
9. **Representative SDC:** the single season's SDC closest to the median SDC.

**Selection — which HRUs get a derived curve vs. the fallback:** apply the paper's
six criteria in order, plus the similarity gate:

| # | Criterion | Data source |
|---|---|---|
| 1 | Full SNODAS coverage (drop border/partial) | fabric vs SNODAS extent (interior Oregon: all pass) |
| 2 | Receives snow (SWE > 0 at some point) | daily series |
| 3 | ≥ 25 SNODAS grid cells | **cells-per-HRU from the gdptools weight matrix** (§8) |
| 4 | < 50 % water-body area | HRU water fraction (repo depstor/water data; §8) |
| 5 | SCA ≥ 50 % (snow over enough of the HRU) | daily SCA |
| 6 | Not constant (SCA not flat for > 80 % of SWE) | curve shape |
| 7 | Similarity ≤ 0.15 | step 8 |

10. **Classification (diagnostic, from the paper):** SCA at `swe_n = 0.5` →
    low (< 45 %) / mid (45–55 %) / high (> 55 %). Recorded as a column; not
    required for the parameter but cheap and useful for QA.
11. **Emit:** `snarea_curve` (11 values × nHRU), `hru_deplcrv` (1:1 index),
    `sdc_status` (`derived` | `default_<reason>`), and diagnostics (similarity,
    reversal count, class, peak-SWE date, melt duration). HRUs failing any
    criterion get the **NHM default depletion curve**, flagged.

## 5. Output

- Per-HRU merged parameter table under `{data_root}/{fabric}/params/merged/`
  (matches the repo's merged-param convention), keyed by the fabric `id_feature`.
- `snarea_curve` serialized in the layout the fabric's PRMS parameter export
  expects (11 ordered values per HRU); `hru_deplcrv` as the per-HRU curve index.
- `sdc_status` + diagnostics travel alongside for QA and are not written to the
  model parameter file itself.

## 6. Validation & testing

- **Harness oracle:** the SNODAS adapter's `swe` output must reproduce the
  existing `snodas_<year>_agg.nc` SWE (Oregon) within a small tolerance — a
  correctness check on the ported harness (units, CRS, masked_mean, weights).
- **Unit tests** (`tests/`, CI-gated — never on the head node):
  - `test_aggregate_*`: adapter/driver on a tiny synthetic grid + 2–3 polygons,
    including the `scov` fill-mask/fraction logic.
  - `test_snarea_curve`: reversal removal, normalization, 11-point interpolation,
    similarity, representative selection, each selection criterion, and the
    default-curve fallback — on small hand-built series with known answers.
- **Oregon end-to-end sanity:** monotonic non-increasing curves; high-SCA
  dominance (paper: ~95 % high); Cascades HRUs derived, low-desert HRUs
  defaulted; spot-check a few curves against their raw daily series.

## 7. Integration notes

- **HRU id join.** The Oregon fabric is the same in both repos (both
  "model_layers 9", 16,814 HRUs) but keyed `hru_id` here vs `nhm_id` in the
  aggregate. Both are contiguous 1..16814 — **verify the mapping is identity (or
  build a crosswalk) once** before trusting the join. Our standalone harness
  aggregates against the gfv2-params fabric directly (`hru_id`), so downstream
  params are natively keyed `hru_id`; the `nhm_id` aggregate is only an oracle.
- **CONUS scale.** CONUS gfv2 is 361,471 HRUs × ~7,670 days. Aggregation is
  per-year and batched (weights cached once); Stage 2 is per-HRU and
  embarrassingly parallel. Follow the repo's SLURM array + afterok-merge DAG;
  set memory per the repo's CONUS guidance.
- **Docs.** Per repo policy, update `docs/`, `README.md`, and
  `slurm_batch/RUNME.md`/`HPC_REFERENCE.md` on the implementation branch.

## 8. Judgment calls (defaulted; revisit if desired)

1. **SCA normalization:** normalize SCA by its value at peak SWE (`sca_at_peak`),
   rather than leaving it raw. Keeps `sca_n ∈ [0,1]` and matches the paper's
   "normalized SCA" language used by the classification thresholds.
2. **≥25-cell count:** take cells-per-HRU from the gdptools weight matrix
   (already computed), not a separate rasterization.
3. **Water fraction (criterion 4):** source from the repo's existing depstor /
   waterbody data if readily available for the fabric; if not, this criterion
   can be a no-op for Oregon and wired in for CONUS. To be finalized in the plan.

## 9. Out of scope (YAGNI)

- Curve clustering / a reduced curve library (decided: one curve per HRU;
  clustering is a possible later follow-up if CONUS param size warrants).
- The paper's climate-relation figures (Daymet tmin/tmax variance) — the harness
  is built to make a Daymet adapter easy later, but it is not part of this work.
- Reproducing the paper's exact 2004–2013 window or its published figures.
